What are tokens in this context?

A named value that replaces a raw value

A token is a named value that replaces a raw value. Instead of #FF3F66 scattered across 40 CSS rules, you have one name — --color-accent — and everything references that name. The token is the name.

--color-bg, --color-text, --color-accent — those are your tokens.

In CSS, it looks like this:

:root {
  --color-accent: #FF3F66;
}

button {
  background: var(--color-accent);
}

h1 {
  color: var(--color-accent);
}

Without the token, you’d have #FF3F66 written out in 40 places. Change the brand color? Find and replace 40 times, hope you missed nothing. With the token, you change it in one place — :root — and everything updates.

The token is --color-accent. The raw value is #FF3F66. You never write the raw value directly. That’s the whole game.

What is a magic value?

A raw value written directly in code — with no name, no explanation, no connection to anything else — is called a magic value (sometimes magic number). It just appears. Like magic. No context, no chain. 

/* magic value — why this color? what is it for? */
color: #e53e3e;

/* token — intention is clear, value lives in one place */
color: var(--color-error);

The “magic” is not a compliment. It means the value is opaque, disconnected, and fragile. Change your error color and you have to find every #e53e3e scattered across the codebase. There’s no chain to pull.

Tokens exist precisely to eliminate magic values. Every raw value gets a name, every name has a home, and nothing is written twice.

The two-layer model

There’s an important distinction between a token and a text style. A token is atomic — one name, one value. A text style is composite — it bundles multiple tokens together. They’re not the same thing, even though designers sometimes use “token” loosely to mean both.

Layer 1 — Tokens (atomic: one name, one value)
font-family: Recursive
font-size-lg: 52px
font-weight-bold: 700
Layer 2 — Text styles (composite: a combination of tokens)
Heading/lg/Bold → Recursive, 52px, 700

In Layer 1: 52px is the raw value. font-size-lg is the token. You never write 52px directly anywhere — you always write font-size-lg. That indirection is the whole point.

Common confusion

Designers often say "token" when they mean "text style." These are not the same thing.

A text style uses tokens. It is not itself a token.

font-size-lg
A token. One name, one value.
Heading/lg/Bold
A text style. A bundle of tokens.

When someone says "I tokenized my typography" they usually mean they created text styles — which is good, but it's one level above the tokens themselves.

What about Figma variables?

When you define variables in Figma — a Name and a Default value — you are declaring tokens. Same concept, different word.

Figma variable: Family Sans → Inter
Token. One name, one value.
CSS custom property: --font-family-sans: Inter
Token. One name, one value.

Figma also has styles (like text styles, color styles). Those are the composite layer — they bundle variables together. So Figma has both layers too, it just uses different words: variables instead of tokens, styles instead of text styles.

You'll also see groups in Figma with names like Primitives, Base, Global, or Foundation. These are just folder labels — organisational words someone chose to group things. They have no technical meaning. What's inside the folder are the tokens. The folder name is irrelevant to what they are.

Why are there so many names for the same thing?

In CSS, --color-accent: #FF3F66 is called a custom property. But everyone, including the MDN docs, also calls them CSS variables. Both names are correct for the same thing.

So now you have:

  • CSS: custom properties = CSS variables = tokens
  • Figma: variables = tokens

The word "variable" appears in both worlds and means the same underlying concept. But "variable" is also a general programming term — it just means "a named thing that holds a value." Which is exactly what a token is.

So token, variable, and custom property are all pointing at the same idea. They're just names from different communities — design systems people say "token", CSS people say "custom property" or "variable", Figma says "variable."

The same concept, three different vocabularies
What it is CSS name Figma name Design systems name
A named value custom property / variable variable token
A bundle of named values — style text style

So they are all variables — why introduce new words?

Why "custom property" instead of just "variable"?

Because in CSS, "property" already has a specific meaning. color, font-size, margin — those are all CSS properties. They're built in, defined by the browser.

When you write --color-accent, you're creating your own property. A custom one. That's why the spec calls it a custom property — it fits precisely into CSS's existing vocabulary. You're extending the property system, not just storing a value.

"Variable" is the informal name that stuck because it's shorter and everyone already knew what a variable was from programming. But technically, in CSS, "custom property" is the correct term.

Why "token"? Where does that word come from?

"Token" comes from the design systems world, not CSS or programming. It was introduced to solve a communication problem between designers and developers.

A variable or custom property describes what it is technically. A token describes what it is for.

  • --color-accent tells you it's a CSS custom property
  • color-accent as a token tells you it's a shared, agreed-upon name that both Figma and CSS use — a contract between design and code

The word "token" signals that this name is system-level — it lives in both worlds and connects them. It's not just a CSS thing or a Figma thing. It belongs to the design system itself.

The short version

Variable
What programmers call it. Generic, informal.
Custom property
What CSS calls it. Technically precise.
Token
What design systems call it. Purpose-driven, cross-tool.

Same thing. Three different angles on it.

Why define color primitives first — why not just name them by purpose?

Because the same raw color can serve many different purposes.

Take Green/500: #14AE5C. Right now it's just a green. But in your system it might be used as the success state color, a button background, a positive trend indicator in a chart, or a badge color.

If you named it directly color-success, that works — until the day your brand changes and success becomes teal instead of green. Now you have to hunt down every place color-success was used and decide if it should change.

Every level is tokens

"Primitive" doesn't mean "not a token." It just means the first level of tokens — the raw ones, the ones that don't point at anything else yet. You could call them basic tokens, or start tokens. They are still tokens.

The system has two levels of tokens, each with a different job:

Level 1 — Primitive tokens (what the color is)
Green/500 → #14AE5C
Just a color. No meaning yet. No purpose. Close to what a CSS custom property does in its simplest form.
Level 2 — Semantic tokens (what the color is for)
color-success → Green/500
color-cta-background → Green/500
Now it has meaning. It points to a primitive. This is where design decisions live.

The pointing is everything

A semantic token doesn't hold a raw value — it holds a reference to another token. That chain is what makes the system flexible. If you want to change your success color from green to teal, you change it in one place — the semantic token. The primitive stays untouched. All components that use color-success update automatically.

If you had skipped primitives and named everything directly, you'd have no stable layer to point to.

The simplest way to remember it

Primitive tokens answer what. Semantic tokens answer why. Same token concept. Two different jobs.

"Semantic" just means "relating to meaning." Green/500 has no meaning on its own. color-success has meaning — it tells you why that color exists in your system.

Tailwind is a token system — just not yours

Once you understand tokens, Tailwind suddenly makes sense. Tailwind is not a different idea — it's the same idea, taken to its logical extreme.

Someone built a complete token system — a full set of named values for spacing, color, typography, borders, shadows, everything — and then generated a utility class for every single one. Instead of writing font-size: var(--font-size-lg) in CSS, you write text-lg directly in HTML. The token is baked into the class name.

The parallel is exact

Your token system
--font-size-lg: 52px defined in :root, used via var(--font-size-lg) in CSS
Tailwind
font-size-lg: 52px defined in Tailwind's config, used via text-lg in HTML

Same indirection. Same one-place-to-change principle. Tailwind just moves the usage point from CSS into HTML, and pre-generates all the classes so you rarely write CSS at all.

The trade-off of naming

Whether you use your own tokens or Tailwind's, the trade-off is the same. A class like body-lg or text-lg is intention-readable — it tells you what it is. But it is also value-opaque — you can't see the actual size without looking it up. A raw value like 20px is the opposite: the value is right there, but the meaning is invisible.

This is why token systems always need two things together: the token definitions as the source of truth, and documentation or Figma as the readable reference. After a while you stop thinking text-lg = 20px and start thinking text-lg = comfortable reading size. That's the goal — thinking in scale, not pixels.

The key difference

Tailwind's tokens are someone else's decisions. Your token system is your own. Tailwind works well until your design needs something it didn't anticipate. Understanding tokens means you know exactly what Tailwind is doing — and when to go beyond it.

Tailwind in the real world — what this means for designers

A lot of agencies use Tailwind, and for good reason. Every project has different developers, different timelines, different codebases. Building a custom token system from scratch every time is expensive. Tailwind gives them a solid, pre-built token system that every developer already knows. Onboarding a new developer? They know Tailwind. No documentation needed.

As a designer, "we're using Tailwind" is something you'll hear in project meetings. Knowing what it means changes how you work.

When working with a Tailwind project

Your Figma tokens need to match Tailwind's scale — or the handoff breaks. If your design uses 16px spacing but Tailwind's nearest step is 14px or 20px, a developer will either bend the design or go off-script and customise Tailwind's config. Knowing this in advance saves a lot of friction. Design to the scale that exists in the codebase, or agree up front that the config will be customised.

Tailwind is configurable

Agencies that care about design quality don't always use Tailwind's default values. They customise Tailwind's config file with their own — their own spacing scale, their own type scale, their own color palette. The framework stays Tailwind, but the tokens become theirs. This is the middle ground between a fully custom system and pure off-the-shelf.

When this happens, your Figma variables should mirror that custom config exactly. The token names in Figma and the token names in Tailwind's config should match. That alignment is what makes design-to-code handoff smooth.

The practical takeaway

Always ask early: are you using Tailwind's default scale, or a custom config?

If the team uses Tailwind's default scale, your Figma variables should mirror the values Tailwind ships with — space-4 = 16px, text-lg = 18px, and so on. You don't invent your own scale. You match theirs.

If the team uses a custom Tailwind config, someone has already defined their own values in that config file. Your Figma variables should mirror those custom values exactly — same names, same numbers.

Either way, your Figma variables are not free to be anything you want. They need to reflect what's actually in the code. If they don't match, design and code drift apart and every handoff becomes a negotiation. If no one has asked that question yet — ask it. It's a design systems decision, and it's yours to care about.

A real example: joining a project using shadcn — no Figma file

You get called into a project. The team is already building with shadcn/ui. No Figma file exists. They need design solutions. Where do you start?

This is a situation where understanding tokens is not theoretical — it's the only way to work correctly.

What is shadcn?

shadcn/ui is a component library built on top of Tailwind. But it adds something important: it uses CSS custom properties for its core design tokens, especially colors. So it is a layer on top of Tailwind — not plain Tailwind.

The shadcn color system looks like this in CSS:

--background: 0 0% 100%;
--foreground: 222.2 84% 4.9%;
--primary: 222.2 47.4% 11.2%;
--primary-foreground: 210 40% 98%;

These are CSS custom properties — tokens — that shadcn components use internally. When a button changes color, it is because --primary changed. Not a Tailwind class. Not a hex value. A token.

So do all your Figma tokens have to be Tailwind tokens?

No — because you are actually dealing with two layers at once, and they work differently:

Layer 1 — shadcn's semantic tokens
The CSS custom properties above. These control the actual component appearance — backgrounds, borders, foreground colors. These you must match in Figma, because changing a button color means changing --primary, not a Tailwind class. These become your Figma variables.
Layer 2 — Tailwind's scale
Spacing, typography, sizing. Layout and component dimensions follow Tailwind's steps. These you should also respect in Figma, so your spacing and type decisions translate cleanly into code.

The workflow

Starting from scratch on a shadcn project without a Figma file, this is the order of operations:

  1. Open the shadcn CSS variables — those become your Figma color variables. Match them exactly: same names, same values.
  2. Check the Tailwind config — default or custom? That becomes your spacing and type scale in Figma.
  3. Build your Figma system to mirror both layers. Now your design tokens and the codebase speak the same language.

Note

There is an official shadcn Figma community file. But starting without it and mapping the tokens yourself is a valuable exercise — it forces you to understand what is underneath the components, rather than just copying styles. Once you have done it once, you understand what any component library is doing under the hood.

Figma and CSS are two representations of the same system

When you define primitives and semantic tokens in Figma, you need to do the exact same work in CSS. Not similar work — the same work. Same layers, same names, same structure.

Figma is not the source that generates CSS. They are both expressions of one shared system, each in their own language.

Figma CSS
Collection: primitives /* primitives */ block in :root
Collection: semantic color /* semantic color */ block in :root
neutral/50 → #fcfcfc --neutral-50: #fcfcfc
bg/page → neutral/50 --color-bg-page: var(--neutral-50)

In CSS, both layers live in :root, separated by a comment:

:root {

  /* primitives */
  --neutral-50: #fcfcfc;
  --neutral-100: #fcfcfd;
  --neutral-200: #f6f6f9;
  /* ... all primitive steps */

  /* semantic color */
  --color-bg-page: var(--neutral-50);
  --color-action-default: var(--cherry-punch-500);
  --color-action-hover: var(--cherry-punch-600);
  --color-action-pressed: var(--cherry-punch-700);
  /* ... */

}

The primitives block is never used directly in your components. You never write var(--neutral-50) on a button. Only semantic tokens get used in components. The primitives just sit there as the stable foundation that semantic tokens point to.

Why this matters

If you skip the primitive layer in CSS and write --color-bg-page: #fcfcfc directly, you don't actually have a two-layer system — you just have a list of semantic tokens with hardcoded values. The chain is broken. Change neutral/50 in Figma and you still have to hunt down every hex value in CSS manually.

In professional teams, tools like Style Dictionary or Tokens Studio exist to keep Figma and CSS in sync automatically — so you don't have to maintain the same structure twice by hand. Emerging approaches using MCP (Model Context Protocol) go further — an AI agent with access to both Figma and the codebase can potentially sync tokens in real time, without a build pipeline. This is still early, but it's the direction things are moving.

Either way: understand the manual structure first. Knowing exactly what the sync should look like is what makes any tool — current or future — understandable and controllable.
